## ⚡ 문서 유형별 템플릿

### API 문서 구조
```markdown
## 엔드포인트명

### 설명
간결한 기능 설명

### 요청
- **URL**: `/api/v1/resource`
- **Method**: `GET`
- **Headers**: `Authorization: Bearer {token}`

### 파라미터
| 이름 | 타입 | 필수 | 설명 |
|------|------|------|------|
| id | string | ✅ | 리소스 ID |

### 응답
\`\`\`json
{
  "status": "success",
  "data": { ... }
}
\`\`\`

### 에러 코드
| 코드 | 설명 |
|------|------|
| 400 | 잘못된 요청 |
| 401 | 인증 필요 |
```

## 📝 테크니컬 라이팅 원칙

### 5C 원칙
| 원칙 | 의미 | 적용 |
|------|------|------|
| **Clear** | 명확함 | 전문용어 정의, 짧은 문장 |
| **Concise** | 간결함 | 불필요한 수식어 제거 |
| **Correct** | 정확함 | 기술적 검증 필수 |
| **Consistent** | 일관성 | 용어/형식 통일 |
| **Complete** | 완전함 | 필요한 정보 누락 없이 |

### 문장 작성 가이드라인
```
✅ 좋은 예:
   - "버튼을 클릭하세요" (능동형)
   - "데이터가 저장됩니다" (직접적)
   
❌ 나쁜 예:
   - "버튼이 클릭되어야 합니다" (수동형)
   - "데이터 저장이 수행됩니다" (우회적)
```
