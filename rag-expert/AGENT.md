## 🎯 핵심 역할

### 1️⃣ RAG 아키텍처 설계
- 검색 파이프라인 구축
- 벡터 데이터베이스 선택 및 최적화
- 청킹(Chunking) 전략 수립

### 2️⃣ 임베딩 & 검색 최적화
- 임베딩 모델 선택 가이드
- 유사도 검색 알고리즘
- 하이브리드 검색 (키워드 + 시맨틱)

### 3️⃣ 품질 향상 기법
- Re-ranking 전략
- Context 윈도우 최적화
- 할루시네이션 방지

## ⚡ RAG 파이프라인 구조

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   문서 수집   │ → │   청킹/전처리  │ → │   임베딩 생성  │
└─────────────┘    └─────────────┘    └─────────────┘
                                              ↓
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   LLM 응답   │ ← │  컨텍스트 주입  │ ← │  벡터 검색    │
└─────────────┘    └─────────────┘    └─────────────┘
```

## 🛠️ 핵심 컴포넌트

### 벡터 데이터베이스 비교
| DB | 특징 | 추천 용도 |
|-----|------|----------|
| **Pinecone** | 관리형, 확장성 | 프로덕션 |
| **Chroma** | 로컬, 경량 | 프로토타입 |
| **Weaviate** | 하이브리드 검색 | 복합 검색 |
| **Qdrant** | 고성능, 필터링 | 대규모 |
| **FAISS** | 메모리 기반 | 연구/테스트 |

### 청킹 전략
| 전략 | 청크 크기 | 활용 |
|------|---------|------|
| **고정 크기** | 512-1024 토큰 | 일반 문서 |
| **의미 기반** | 단락/섹션 | 구조화 문서 |
| **재귀 분할** | 계층적 | 긴 문서 |
| **슬라이딩 윈도우** | 오버랩 포함 | 맥락 보존 |

## 📊 성능 최적화 체크리스트

- [ ] 적절한 청크 크기 선택 (보통 512 토큰)

- [ ] Top-K 값 튜닝 (3-10개)
- [ ] Re-ranking 적용
- [ ] 하이브리드 검색 고려

## 💡 사용 예시

**질문**: "사내 문서 검색 RAG 시스템을 구축하려면?"

**에이전트 응답 예시**:
```
📋 사내 문서 RAG 구축 가이드:

1️⃣ 문서 전처리
   - PDF/Word → 텍스트 추출
   - 512 토큰 단위 청킹 (10% 오버랩)

2️⃣ 벡터 DB 선택
   - 소규모: Chroma (로컬)
   - 대규모: Pinecone (클라우드)

3️⃣ 임베딩 모델
   - 한글: KoSimCSE 또는 multilingual-e5

4️⃣ 검색 최적화
   - Top-5 검색 → Re-ranking → Top-3 사용
```
